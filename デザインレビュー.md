# おくすリング デザインレビュー

レビュー日: 2026-07-13
対象: 服薬リマインダーサービス「おくすリング」
観点: ビジュアル / ライティング / UX・使い勝手 / UI / アクセシビリティ

---

## 総評

まず、コンセプトが非常に良いです。「多機能で複雑になりがちな服薬管理アプリに対して、"飲み忘れ防止"に必要な機能だけへ絞る」という**引き算の設計思想**が、README・LP・aboutページを通じて一本の筋として通っています。プロダクトとして何者なのかがブレていないのは、それだけで大きな強みです。

見た目もやさしいグリーン〜ブルーの世界観で統一されていて、「服薬」「安心」「家族の見守り」というテーマにきれいに合っています。全体としてかなり完成度が高く、細部の配慮も随所に感じられました。

その上で、「あと一歩でグッと良くなる」ポイントを、褒めるところと直したいところに分けて挙げます。

## 👍 良いところ（このまま活かしてほしい）

### 1. コンセプトとコピーが一貫して温かい
- 「飲んだよ」というボタン文言、`例：自分、みーちゃん、ポチ` `例：オレンジジュースと一緒に飲ませる` といったプレースホルダーが具体的で、実際の使用シーンがありありと浮かびます。
- **ペットも服薬管理の対象に含めている**視点はユニークで、ターゲットの解像度の高さが伝わります。
- aboutページの「家庭全体で支え合える仕組み」という締めまで、終始やさしいトーンで書けています。

### 2. デザイントークンの導入
- `:root` でカラーを CSS 変数として定義し、グラデーションもトークン化している姿勢はとても良いです（`--color-primary-gradient` など）。設計者として正しい方向を向いています。

### 3. カード型UIとやわらかい質感
- 角丸20px・淡いシャドウ（`0 8px 24px rgba(...)`）・カード間の余白が心地よく、モバイルでの見やすさに寄与しています。
- `.title-dot`（見出し横の丸ドット）や 💊 アイコンなど、小さな装飾で親しみやすさを出せています。

### 4. ロゴがサービスの世界観を完璧に体現している（講師も高評価）
- お薬（分包）のキャラクターが家族（緑・黄・ピンクの3人）を両手で包み込み、若葉・ハート・カプセル・肉球があしらわれたロゴは、**「家族もペットも、やさしく見守る」というコンセプトそのもの**を一枚で語れています。淡いパステルのトーンもアプリ本体の世界観とぴったり一致していて、完成度が高いです。
- 補足の気づき: ロゴ内のタグライン **「家族も、ペットも、やさしく見守る。」** は、それ自体が優れたキャッチコピーです。上記【中 #7】で指摘した「LPにキャッチコピーが無い」問題は、**このタグラインをテキストとしてLPの主役コピーに据える**だけで解決できます。ロゴにすでに答えがあります。

### 5. 背景色がサービスの世界観に合っている（講師も高評価）
- 背景の `#F5FBFA`（ごく淡いミント／アクアの白）が、**清潔感がありつつ冷たすぎない、"お薬・医療系アプリらしい"心地よいトーン**を作れています。真っ白でも寒色グレーでもない、この絶妙な淡さがサービスの安心感に効いています。
- カード（純白 `#FFFFFF`）との明度差もほどよく、コンテンツが自然に浮き上がって見えます。ベースカラーの選定は成功しています。

### 6. 状態のフィードバックが明快
- 「✔ 済 / 未」「閲覧のみ」「連携済み」など、ステータスがラベルで明示されていて、今どういう状態かが伝わります。

### 7. マイクロインタラクションへの目配り
- タップ時の `transform: scale(0.98)`、ホバー時の `opacity` 変化で、操作の手応えを演出できています。`@media (hover: hover)` でタッチ端末を除外しているのも丁寧です。

### 8. 招待フローの配慮
- 招待ページの 💌、「ログイン後、自動でこのページに戻ります」という予告、`return_to` を使ったログイン後の復帰導線など、招かれた家族の体験がよく考えられています。

### 9. 基盤的な作り込み
- OGP・Twitterカード・favicon・apple-touch-icon まで設定済み。SNS共有時の見え方まで意識できています。
- 削除操作には `turbo_confirm` の確認ダイアログ。破壊的操作への配慮ができています。

## 🔧 直したいところ（優先度順）

### 優先度：高

#### 1. フラッシュメッセージ（notice / alert）が未スタイル
`app/assets/stylesheets/application.css` に `.notice` `.alert` のスタイルが**一切ありません**。さらに `layouts/application.html.erb` では、これらがヘッダーより上・`page-container` の外側にベタ書きで出力されています。

- 「登録しました」「削除しました」などの**成功通知が素の黒文字**で画面最上部に出るだけで、目立たず・レイアウトからも浮きます。
- 成功（緑系）とエラー（赤系）の区別も付きません。

→ 服薬記録という「押したことが伝わる安心感」が命のサービスなので、ここは最優先で整えたいです。カラー・アイコン・数秒で消えるトースト風など、フィードバックとして成立させましょう。

#### 2. 家族の「飲んだ / まだ」がホーム一覧で見えない
コンセプトの核は「家族が**ちゃんと飲めたか**を見守れる」ことなのに、ホームの「家族の予定」欄に出るのは **時刻・名前・タイトル・「閲覧のみ」バッジだけ**で、肝心の服薬状況（済／未）が表示されていません。詳細ページに入らないと分からない状態です。

→ 一番の価値であるはずの「見守り」が一覧で完結しないのはもったいないです。自分の予定と同様に「✔ 済 / 未」を家族欄にも出すことを強くおすすめします。

#### 3. 主要アクションのボタン優先度が逆転している
「飲んだよ」ボタン（`.med-action .app-button`）は**白背景の控えめなスタイル**なのに対し、「＋新しい服薬スケジュールを登録」は色付きの primary ボタンで最も目立ちます。

- 毎日の中心操作は「飲んだよ」の記録のはず。ここが一番目立つべきです。
- 現状は「新規登録」という頻度の低い操作の方が視覚的に強く、優先順位とデザインの強弱が噛み合っていません。

→ 「飲んだよ」を primary 相当の強さに、「新規登録」を一段控えめにすると、日々の使い勝手が変わります。

### 優先度：中

#### 4. CSS の重複定義でデザイントークンが死んでいる
同じセレクタが複数回定義され、後勝ちで意図が上書きされています。

- `.app-button-primary` … 219行目でせっかく**グラデーション**（`--color-primary-gradient` + シャドウ）を当てているのに、633〜637行目で**単色 `#73d8b3`** に上書きされ、グラデーションが効いていません。
- `.app-button` は3箇所、`.med-item` `.family-member` も2箇所ずつ定義。
- `.form-button` は定義されているがどこからも使われていない（デッドコード）。

→ せっかくトークンを整えたのに、上書き合戦で結果がブレています。定義を1箇所に集約し、「意図した見た目＝実際の見た目」を一致させましょう。

#### 5. 色のハードコードが多く、変数が活きていない
`#666` `#777` `#555` `#2f8f5b` などの直値が本文・補足・ステータスに散在し、`--color-text-secondary` `--color-text-muted` などの変数が実際にはほとんど使われていません。

→ トーンの微調整や統一が効かなくなります。グレー系こそ変数に寄せると、全体の色の一貫性が上がります。

#### 6. ベースのタイポグラフィ設定が無い
`body` に `font-family` も `line-height` も指定がなく、見出しの余白もブラウザ既定に依存しています。カードの中は整っていますが、**土台の行間・フォント・見出しリズム**が定まっていません。

→ `body` にベースの `font-family` / `line-height` を設定し、見出しの `margin` を整えるだけで、全体の"きちんと感"が底上げされます。

#### 7. ライティングの細部
- **「飲む人は変更できません。変更する場合は新しく予定を登録してください」**（`_form.html.erb`）— 制約だけが唐突に提示され、理由が分かりません。一言「※記録を正しく残すため」等の補足があると納得感が出ます（可能ならそもそも変更させてあげたい）。
- **「友達」と「友だち」の表記**（通知設定）— LINE公式の表記は「友だち」です。合わせると自然です。
- 「連携状態を更新する」がただのリンク（＝再読み込み）になっており、ボタンの見た目なのに何が起きるか分かりにくいです。「最新の状態を確認する」など動作が伝わる文言＋挙動の明示を。
- LP（未ログイン時）に**キャッチコピーが無い**：ロゴのすぐ下がいきなり「できること」です。「家族みんなの"飲んだ"を見守る」のような1行のヒーローコピーがあると、3秒で価値が刺さります。

#### 8. 誤操作からの復帰導線が無い
「飲んだよ」を押した後は「✔ 済」表示になるだけで、**間違えて押した場合に取り消す手段が見当たりません**。誤タップは日常的に起きるので、「記録を取り消す」導線があると安心です。

#### 9. 再通知設定のUI連動
「再通知を有効にする」チェックが OFF でも「再通知までの時間」セレクトが常に活性・表示のままです。

→ 無効時はセレクトを `disabled`（グレーアウト）にすると、状態と操作が一致して迷いません。

### 優先度：低（細部の磨き込み）

#### 10. 「今日のお薬の予定」に日付が無い
見出しは「今日の」ですが、実際の日付表示がありません。`7/13(日)` のような日付があると、記録の文脈が明確になります。

#### 11. アクセシビリティ
- ハンバーガー `☰`・戻る `＜`・各種絵文字アイコン（💊✅👤🔗💌）に `aria-label` / `aria-hidden` の指定がなく、スクリーンリーダーでの意味付けが弱いです。装飾絵文字は `aria-hidden="true"`、機能ボタンには `aria-label` を。
- コピー完了メッセージ（`.copy-message`）は `hidden` の付け外しで表示されますが、`aria-live` が無いため読み上げ環境に伝わりません。
- 薄いグレー（`.readonly #888`、`--color-text-muted #8A959A`、フッター13px など）や、primaryボタンの `#73d8b3` に白文字は、**コントラスト比が WCAG AA を下回る可能性**があります。特にボタンの緑×白は要チェックです。
- フォーム入力欄は `<h3>` を見出し代わりにしていますが、`<label>` として `for` で紐付いていません。タップ領域・支援技術の両面で `label` 紐付けが望ましいです。

#### 12. コード起因のレイアウトリスク
- `notification_settings/show.html.erb` で、2つ目の `<section>` を **`</div>` で閉じている**箇所があります（タグの対応ずれ）。ブラウザが補正するため今は見えていませんが、レイアウト崩れの温床です。
- `family_links/accept.html.erb` と `accept_l2` `accept_l3` が併存しています。検証用の名残であれば、整理しておくと保守が楽になります。

## 🧑‍🏫 講師からの追加指摘

レビュアー（デザイン講師）が実際に触れて気づいた点を追記します。上のコード観点の指摘とも重なる部分があり、その一致は「見た目でも構造でも同じ課題が出ている」＝優先度が高い証拠です。

### A. Flash に背景色などのデザインを付けたい
（上記 **優先度・高 #1** と一致）
- 現状 `.notice` / `.alert` は無地のテキストのみ。**背景色**（成功＝緑系、エラー＝赤系）・余白・角丸を付けて、"メッセージの箱"として成立させたいです。
- サービスの世界観に合わせるなら、成功は `--color-primary-start` 系の淡い緑背景、エラーは `--color-danger` 系の淡い赤背景が自然です。
- 表示位置も、現状はヘッダーより外・`page-container` の外に出ているので、コンテンツ幅に揃える／画面上部に固定して数秒で消えるなど、置き場所も併せて設計を。

### B. ハンバーガーメニューが小さく、PCだと見逃す
- `summary` の `☰` は文字サイズ既定のままで、当たり判定（クリック領域）も文字ぶんしかありません。特にPCの広い画面では埋もれて気づかれない恐れがあります。
- 対策案:
  - アイコンを一回り大きく（例: `font-size: 24〜28px`）し、`padding` でタップ/クリック領域を 44px 四方程度確保する。
  - 「メニュー」ラベルを併記する、または枠線・薄い背景で"押せる場所"だと分かる見た目にする。
  - `aria-label="メニュー"` を付けて意味も補う（アクセシビリティ #11 とも連動）。

### C. ボタンのルールを整理したい（primary / secondary の使い分け）
（上記 **優先度・高 #3**／**中 #4** と同じ根に繋がる重要指摘）
- 基準を「**DBに保存する＝主要アクションは primary**」「**画面遷移リンク的なものは secondary**」と定義するのは、迷いが消える良いルールです。この基準で棚卸しすると:

| 操作 | 種別 | 望ましいスタイル |
| --- | --- | --- |
| 「飲んだよ」（服薬を記録＝保存） | 主要アクション | **primary** ←現状は控えめで要昇格 |
| 「登録する / 更新する」（保存） | 主要アクション | **primary** |
| 「保存する」（通知設定の保存） | 主要アクション | **primary** |
| 「LINEでログイン」 | 主要アクション | **primary**（現状OK） |
| 「＋新しい服薬スケジュールを登録」 | 画面遷移 | secondary ←現状 primary で目立ちすぎ |
| 「編集」 | 画面遷移 | secondary |
| 「＋家族を招待する」 | 画面遷移 | secondary |
| 「削除 / 共有を解除」 | 破壊的 | danger（控えめ＋赤）※現状の抑えた方針は◎ |

- あわせて、CSS 側の `.app-button` / `.app-button-primary` の**重複定義を1箇所に集約**し、primary＝色付き・secondary＝白背景＋枠線・danger＝赤系、と**3種を明確に定義**しておくと、今後は迷わず貼れます（現状は primary のグラデーションが後段で単色に上書きされて死んでいる点も同時に解消できます）。

### D. フッターを常に画面下部に配置したい（sticky footer）
- 現状 `.site-footer` は `margin: 40px auto 24px` で、**コンテンツ量が少ないページでは画面中ほどに浮いてしまう**恐れがあります（「今日の予定なし」などの短い画面で顕著）。
- 対策は、`body` を縦フレックス（またはグリッド）にして、メイン領域を伸縮させる典型的な sticky footer パターンが手堅いです:
  ```css
  body {
    min-height: 100dvh;      /* モバイルのアドレスバー対策に dvh */
    display: flex;
    flex-direction: column;
  }
  /* yield されるメインコンテンツを包む要素に */
  main { flex: 1 0 auto; }
  .site-footer { margin-top: auto; }
  ```
  - ※現状 `layout` の `<%= yield %>` が `main` などで包まれていないので、フレックスで制御するには**ラッパー要素を1つ用意**する必要があります。ここは構造の小改修とセットで。

### E. 緑ボタンの白文字が読みにくい（コントラスト不足）
（上記 **アクセシビリティ #11** の懸念が、計測で明確にアウトと確定）
- primary ボタンの緑背景に白文字は、**コントラスト比を実測すると WCAG 基準を大きく下回ります**。

| 背景色 | 文字色 | コントラスト比 | 判定（通常テキスト 4.5:1 / UI・大文字 3:1 が必要） |
| --- | --- | --- | --- |
| `#73d8b3`（単色版 primary） | 白 | 約 **1.7:1** | ❌ 大きく不足 |
| `#63D98A`（グラデ開始色） | 白 | 約 **1.8:1** | ❌ 大きく不足 |

- 明るい緑（パステル系）は白文字と明度が近く、**光の反射する屋外や高輝度画面ではほぼ判読不能**になります。ご指摘のとおりです。
- 対策の方向性:
  - **文字色を濃くする**：白ではなく濃い緑系（例: `#1f5f3f` など）にすると、やさしい色味を保ったまま可読性を確保できます。
  - **または背景を濃くする**：ボタンの緑をもう一段濃いトーンに落として白文字を活かす。ただし世界観の淡さは薄れます。
  - どちらでも良いですが、**「淡い背景×濃い文字」か「濃い背景×白文字」のどちらかに寄せる**のが原則。今は「淡い背景×白文字」で最も読みにくい組み合わせになっています。
  - 目標は最低 4.5:1（本文相当）。ボタン文字は視認性が体験に直結するので、ここは妥協せず確保したいポイントです。

### F. ロゴの背景色がサイト背景と一致せず浮いている
- ロゴ自体は世界観にぴったり（上記・良いところ #4）ですが、**ロゴ画像 `logo.png` の背景が純白 `#FFFFFF`**、対してサイト背景は淡いミント `#F5FBFA`。この差のせいで、LP・aboutページでロゴが**白い四角のパネルとして浮いて見え**、せっかくの一体感が損なわれています。もったいないポイントです。
- 対策（おすすめ順）:
  1. **背景を透過（transparent）にしたPNGを書き出す** ── どんな背景色に変えても馴染むので最も汎用的。今後の配色変更にも強いです。これが本命。
  2. ロゴの背景を**サイトと同じ `#F5FBFA` で書き出す**。手軽ですが、将来サイト背景を変えると再び浮きます。
  3. CSS 側で対処するなら、ロゴを置く領域（`.lp-logo` / `.about-logo`）の背景を白にして「白の上に置くことにする」方法もありますが、淡いミント背景の良さを消してしまうので非推奨。
- ファビコン・apple-touch-icon・OGP画像も同じ素材なら、透過版／背景色を揃えて書き出しておくと全体で統一できます。

### G. ボタンが横に大きすぎてバナーに見える（横幅の制限）
- `.app-button` が `display: block; width: 100%;` のため、**PCの広い画面ではボタンが画面幅いっぱいに間延びし、押せるボタンというより"バナー画像"のように見えてしまいます**。ご指摘のとおりです。
- モバイル前提の全幅ボタンは自然ですが、そのままPCに持ち込むと横長になりすぎます。対策:
  - コンテンツ自体は `page-container`（`max-width: 640px`）などで制限されていますが、その中でもボタンだけは**最大幅を設けて中央寄せ**すると収まりが良くなります。
    ```css
    .app-button {
      max-width: 360px;   /* 例。ボタンとして自然な横幅に */
      margin-left: auto;
      margin-right: auto;
    }
    ```
  - あるいは、**文字量に応じた幅（`width: auto` + `padding` 左右多め）＋中央寄せ**にして、ボタンを「内容ぶんの大きさ」にする方向も。現に「飲んだよ」ボタンは `width: auto; min-width: 96px` で良い塩梅なので、**この考え方を主要ボタン全体のルールに広げる**とまとまります。
  - この横幅ルールは【ボタンのルール整理 C】とセットで、「primary/secondary/danger の色」＋「共通の最大幅・寄せ」まで決めてしまうと、ボタン設計が一気に安定します。

### H. チェック（済）アイコンをアイコンフォントに置き換えると可愛くなる
- 現状の「✔ 済」「✅」は**環境依存の絵文字・文字**で表示しています。これには2つの弱点があります。
  1. **OS/ブラウザで見た目が変わる**（iOS・Android・Windows で ✔ や ✅ の形・色がバラバラ）。デザインをコントロールできません。
  2. サービスのやわらかい世界観に対して、絵文字のチェックは少し素っ気なく、質感が浮きがちです。
- ご提案のとおり、**Google の Material Symbols（Material Icons）などのアイコンフォント／SVGに置き換える**と、
  - 全環境で**同じ形・同じ色**で表示でき、
  - `check_circle` のような**丸みのあるアイコン**を選べば、パステルな世界観に合った可愛い仕上がりになります。
  - 色もCSSで自在（`--color-primary-start` の緑に合わせるなど）にでき、状態（済＝緑の塗り／未＝グレーの輪郭）で対比も付けやすいです。
- 対象箇所: ホームの「✔ 済」、詳細ページの「✔ 済 / 未」、LP機能紹介の 💊✅👨‍👩‍👧、家族一覧の 👤、招待の 🔗💌 など。**まずは服薬状態を表す「済/未」から**アイコン化すると効果が大きいです。
- 補足（実装面）: Material Symbols は Web フォント（外部CDN）または SVG で導入できます。読み込み方法やライセンス表記の要否だけ確認しておくと安心です。装飾アイコンには `aria-hidden="true"` を付ける運用も、アクセシビリティ #11 と合わせて。

### I. 利用規約・プライバシーポリシーが横幅無制限で読みにくい
- `terms.html.erb` / `privacy.html.erb` は、**`<p>` `<h3>` `<hr>` が `<body>` 直下に直接置かれていて、幅を制限するラッパー要素がありません**。そのためPCの広い画面では本文が**画面端から端まで一行に伸び**、視線の折り返しが長くなって非常に読みにくい状態です。ご指摘のとおりです。
- 面白いことに、aboutページには `.about-page`（`max-width: 800px; margin: 0 auto`）が付いているのに、**規約系のこの2ページだけ抜け落ちています**。おそらく付け忘れです。
- 対策:
  - この2ページを `.page-container`（既存・`max-width: 640px`）などのラッパーで囲むのが最短。長文の可読性を優先するなら、専用の `.legal-page`（`max-width: 720〜800px` 程度）＋中央寄せを用意しても良いです。
    ```erb
    <div class="legal-page">
      ... 本文 ...
    </div>
    ```
  - 併せて、長文なので **`line-height: 1.8` 前後・段落間の余白**を確保すると、規約特有の読みづらさが和らぎます（土台のタイポグラフィ整備【中 #6】とも連動）。
- 補足: `<hr>` を各見出し直後の区切りに使っていますが、見出し＋余白だけでも十分区切れます。装飾として残すなら、色を薄く（`--color-border`）してトーンを揃えると上品になります。

### J. カード上部のマージンがサイト全体で大きすぎる（共通の症状）
> 当初は「家族共有設定」ページで気づいた症状ですが、**サイト全体のカードで共通して起きている**問題です。ほぼ全てのカードが「見出し始まり」のため、同じ過剰マージンが全ページに乗っています。1箇所直せば全ページ一括で改善するので、**優先度は中〜高**として扱う価値があります。

- カード（`.app-card`）の上部に不自然な余白ができています。原因は**マージン相殺の不発**です。
  - カードの最初の子が**プレーンな `<h2>家族一覧</h2>`**（クラス無し）で、ブラウザ既定の上マージン（約 `0.83em`）が付きます。
  - `.app-card` には `padding: 24px` があり、**パディングはマージン相殺を止める**ため、`h2` の上マージンがカードの内側パディング **24px の上にそのまま加算**されます。
  - 結果、カード上端〜「家族一覧」の間に「24px＋約20px」の**約45pxの過剰な余白**が生まれます。
- これは【中 #6 ベースのタイポグラフィ設定が無い】の具体的な症状です。CSSリセット／見出しマージンの正規化が無いことが根本原因なので、直すと**他ページのカードや見出しの余白も一斉に整います**（通知設定など、プレーン `<h2>` を使う画面すべてに効きます）。
- 対策（どちらか）:
  ```css
  /* カード内の先頭要素の上マージンを消す（ピンポイント） */
  .app-card > :first-child { margin-top: 0; }
  ```
  ```css
  /* もしくは見出しのマージンを土台からルール化（推奨） */
  h2, h3 { margin-top: 0; }   /* 必要な余白は個別クラスで付与 */
  ```
  - `.home-title` `.schedule-card-title` `.lp-section-title` などは個別にマージン指定済みなので、**プレーンな見出しの既定マージンを正規化する**方針が全体最適です。

### K. BOT未連携のうちはトップに「通知設定」への導線を出したい（オンボーディング）
- このサービスの生命線は「服薬時間に**通知が届く**こと」ですが、その通知を有効にする **LINE BOT連携は、ハンバーガーメニュー →「通知設定」の奥**にあります。気づかないまま進む導線です。
- 結果として、**BOT未連携のままスケジュールだけ登録 → 通知が一切来ない → 「使えない」と誤解して離脱**、という最悪のパターンが起こりえます。中核価値が届かないのは非常にもったいないです。
- ご提案のとおり、**未連携時（`current_user.line_bot_connected?` が false）だけ、トップページ上部に目立つ導線を出す**のが効果的です。実装のヒントになる材料はすでに揃っています（`line_bot_connected?` 判定は通知設定ページで使用済み）。
  - 例: ホーム上部に「⚠️ まだ通知を受け取れません。BOTを連携すると服薬時間にお知らせします → [通知設定へ]」といった**注意喚起バナー**を表示。
  - 連携が済んだら自動で消える（条件表示）ので、既存ユーザーの邪魔にはなりません。
  - この"バナー"は【Flash／メッセージのデザイン A】で整える背景色付きの箱を流用でき、デザイン投資を一石二鳥にできます。
- さらに一歩進めるなら、**初回ログイン直後にBOT連携を促すステップ**（オンボーディング）を挟むと、そもそも未連携状態を作らせない設計にできます。まずはトップの導線から着手するのが手軽で効果大です。

### L. 「こんにちは、◯◯さん」がPCで左上に浮いて見える
- ホームの挨拶（`.home-greeting`）は、**`.page-container` の外側**に置かれています。
  - `.page-container` は `max-width: 640px; margin: 0 auto`（中央寄せ）＋左右パディングを持つのに対し、`.home-greeting` は `margin: 24px 0 16px` のみで**max-widthも中央寄せも左右パディングも無い**。
  - そのためPCの広い画面では、**挨拶だけが画面の左端にへばりつき**、その下の中央寄せされたコンテンツ（タイトル・カード）と**左端が揃わず**、ぽつんと浮いて見えます。ご指摘のとおりです。
- 対策（どちらか）:
  1. **挨拶を `.page-container` の中に入れる**（構造で解決・おすすめ）。タイトルやカードと左端が自然に揃います。
  2. もしくは `.home-greeting` に `page-container` と同じ**最大幅・中央寄せ・左右パディング**を与える。
    ```css
    .home-greeting {
      max-width: 640px;
      margin: 24px auto 16px;
      padding: 0 20px;   /* page-container と揃える */
    }
    ```
- 補足: この「コンテナの外に要素が出ていて幅・位置が揃わない」問題は、【I 規約ページ】【J カード余白】と同じ"**コンテナ／余白ルールが統一されていない**"という根を共有しています。コンテンツ全体を包む共通レイアウト（`main` ＋最大幅ルール）を一度決めてしまうと、この種のズレがまとめて解消します。

### M. ファビコンが低解像度で何か分からない
- 現状のファビコンは**ロゴをそのまま縮小した**ものです。ロゴは細部（分包キャラの顔・家族・若葉）まで描き込まれているため、**16px まで縮むとディテールが完全につぶれ**、「緑の丸に白いモヤっとした塊」にしか見えません。何のサービスのタブか判別できず、ブラウザのタブやブックマークで埋もれてしまいます。ご指摘のとおりです。
- ファビコンは**ロゴの縮小版ではなく、専用にデザインする**のが原則です。ポイント:
  - **1つの概念だけ**に絞る（要素を詰め込まない）。
  - **太い線・大きな面・高コントラスト**で、小さくても形が成立するように。
  - 余白を取りすぎず、**モチーフを画面いっぱい**に。
- モチーフ案:
  - サービス名「おくす**リング**」にかけて、**リング（輪）＋錠剤／カプセル**を組み合わせた単純な記号。名前とも意味が繋がり覚えやすいです。
  - あるいは**カプセル1粒**だけ、または**分包キャラの顔だけ**を思い切り大きく・単純化した形。
  - 色は今のミント／グリーンを踏襲すればブランドの一貫性は保てます。
- 確認方法: 実際に **16×16 / 32×32 に縮小して視認**し、それでも一目で分かるかをチェック。apple-touch-icon（ホーム画面用・180px程度）は少し情報量を足せるので、**サイズ別に作り分ける**とより丁寧です。
